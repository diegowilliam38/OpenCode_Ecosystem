Este documento foi escrito para atender simultaneamente a dois públicos: o \textbf{leitor iniciante}, que está tendo seu primeiro contato com o ecossistema OpenCode, e o \textbf{pesquisador avançado}, que busca fundamentação rigorosa para cada decisão arquitetural. Para conciliar estas duas necessidades, cada seção segue uma estrutura didática em três camadas:

\begin{enumerate}[itemsep=2pt]
  \item \textbf{O Que É} (Nível Introdutório): Explicação conceitual em linguagem acessível, com analogias do cotidiano. Para leitores sem conhecimento prévio.
  \item \textbf{Por Que Foi Feito} (Nível Intermediário): Justificativa da decisão arquitetural, conectando o ``o quê'' ao ``por quê''. Para leitores com conhecimento técnico básico.
  \item \textbf{Como Foi Implementado} (Nível Avançado): Detalhamento técnico com código, formalismo matemático e citações acadêmicas com DOI. Para pesquisadores e desenvolvedores.
\end{enumerate}

\subsection*{Glossário de Termos para o Leitor Iniciante}

Antes de prosseguir, familiarize-se com estes termos que aparecerão ao longo do texto:

\begin{itemize}[itemsep=1pt]
  \item \textbf{Agente}: Um programa de computador especializado em uma tarefa específica (como pesquisar artigos, escrever textos ou revisar qualidade). Pense nele como um ``funcionário virtual'' com uma função bem definida.
  \item \textbf{Pipeline}: Uma sequência de etapas automatizadas, como uma linha de montagem. No nosso caso, o pipeline SEEKER $\rightarrow$ MASWOS $\rightarrow$ PhD Auditor transforma uma ideia de pesquisa em um artigo científico completo.
  \item \textbf{API} (Interface de Programação de Aplicações): Um ``cardápio'' que um site ou serviço oferece para que programas de computador possam consultar seus dados. Por exemplo, a API do Banco Mundial permite que um programa obtenha o PIB do Brasil sem precisar ``navegar'' no site como um humano faria.
  \item \textbf{DOI} (Digital Object Identifier): Um código único e permanente que identifica um artigo científico, como um ``CPF'' do artigo. Exemplo: \texttt{10.1073/pnas.36.1.48}.
  \item \textbf{JSON} (JavaScript Object Notation): Um formato de arquivo que armazena dados de forma estruturada, como uma ficha de cadastro digital.
  \item \textbf{Hash SHA-256}: Uma ``impressão digital'' matemática de um texto. Se uma única letra do texto mudar, o hash muda completamente. Usado para garantir que ninguém alterou um registro.
  \item \textbf{Token}: A menor unidade de processamento de um modelo de linguagem (aproximadamente 3/4 de uma palavra em português). O modelo big-pickle processa até 200.000 tokens por vez --- equivalente a cerca de 150.000 palavras, ou um livro de 300 páginas.
  \item \textbf{MCP} (Model Context Protocol): Um protocolo padronizado (criado pela Anthropic em 2024) que permite a agentes de IA usar ferramentas externas de forma segura, como um ``controle remoto universal'' para serviços.
\end{itemize}

\section{Introdução: O Que É o OpenCode Ecosystem e Por Que Ele Existe}
% ══════════════════════════════════════════════════════════════════════════

\subsection{O Que É (Nível Introdutório)}

Imagine que você precisa escrever um artigo científico de alta qualidade --- digamos, sobre a relação entre investimento em pesquisa e desenvolvimento (P\&D) e inovação no Brasil. Você precisaria: (a) buscar dezenas de artigos em bases como arXiv, Google Scholar e PubMed; (b) ler e fichar cada um deles; (c) obter dados socioeconômicos do Banco Mundial, IBGE e OCDE; (d) escrever o artigo seguindo normas ABNT; (e) revisar a qualidade do texto; (f) verificar se todas as citações estão corretas; (g) submeter a periódicos Qualis A1.

O \textbf{OpenCode Ecosystem} automatiza todo esse processo. Ele é uma ``universidade dentro do terminal'' --- 125 agentes especializados (como pesquisadores, escritores, revisores e estatísticos) que trabalham juntos, orientados por um orquestrador central, para entregar resultados com o mesmo rigor de um artigo publicado nos melhores periódicos.

\subsection{Por Que Foi Feito (Nível Intermediário)}

A pesquisa acadêmica enfrenta três desafios estruturais que este ecossistema resolve:

\begin{enumerate}[itemsep=2pt]
  \item \textbf{Fragmentação de fontes}: Dados relevantes estão dispersos em dezenas de APIs diferentes (arXiv, Semantic Scholar, Banco Mundial, Yahoo Finance, PubMed). Um pesquisador gastaria dias apenas para aprender a usar cada uma delas. O \textbf{DataOrchestrator} resolve isso: você pergunta ``PIB do Brasil'' e o sistema descobre automaticamente qual API usar.
  \item \textbf{Rastreabilidade insuficiente}: Artigos tradicionais listam referências, mas não permitem verificar se cada afirmação realmente corresponde ao que a fonte diz. O \textbf{Sistema de Auditoria Caixa Branca} resolve isso: cada parágrafo é vinculado à sua evidência original com hash criptográfico.
  \item \textbf{Viés metodológico}: Pesquisadores tendem a usar sempre o mesmo tipo de raciocínio (por exemplo, apenas análise estatística, sem considerar aspectos estratégicos). O \textbf{Reasoning Orchestrator} resolve isso: 68 tipos de raciocínio --- incluindo 10 da Teoria dos Jogos --- são aplicados sistematicamente.
\end{enumerate}

\section{Alteração 1: PyPI Scout --- Como o Ecossistema Descobre Bibliotecas}
% ══════════════════════════════════════════════════════════════════════════

\subsection{O Que É (Nível Introdutório)}

O \textbf{PyPI} (Python Package Index, pronunciado ``pai-pi-ai'') é o repositório oficial de bibliotecas da linguagem Python --- como uma ``loja de aplicativos'' para programadores. Existem mais de 500.000 bibliotecas disponíveis, e encontrar a certa para cada tarefa é como procurar uma agulha no palheiro.

O \textbf{PyPI Scout} é o ``bibliotecário digital'' do ecossistema. Ele mantém um catálogo curado das 22+ bibliotecas mais úteis para pesquisa acadêmica, organizadas por tema (artigos, dados mundiais, finanças, biologia, etc.) e com uma ``nota de compatibilidade'' (matriz de afinidade) que indica quão bem cada biblioteca funciona com cada pipeline do ecossistema.

\subsection{Por Que Foi Feito (Nível Intermediário)}

Antes do PyPI Scout, o ecossistema usava uma técnica chamada \textit{scraping HTML} --- basicamente, ``lia'' a página web do PyPI como um humano faria, extraindo informações do código da página. Em maio de 2026, o PyPI implementou uma proteção chamada \textit{Client Challenge} da Cloudflare, que exige que o navegador execute JavaScript para provar que não é um robô. Como o scraper não executava JavaScript, ele foi bloqueado.

A solução foi migrar para a \textbf{API JSON oficial} do PyPI. Uma API é como um ``cardápio digital'' --- em vez de ``ler'' a página inteira, o programa faz um pedido específico (``me dê os dados do pacote \texttt{wbgapi}'') e recebe exatamente o que pediu, em formato estruturado.

Esta transição ilustra o que o filósofo da ciência Thomas Kuhn chamou de ``mudança de paradigma'' (KUHN, 1962): quando uma abordagem estabelecida encontra anomalias que não consegue resolver, um novo paradigma emerge.

\subsection{Como Foi Implementado (Nível Avançado)}

\subsubsection{Catálogo Curado}

O arquivo \texttt{opencode\_catalog.json} armazena metadados de 22+ bibliotecas em 6 categorias. Cada entrada contém:

\begin{itemize}[itemsep=1pt]
  \item \texttt{name}: Nome do pacote no PyPI
  \item \texttt{install}: Comando de instalação (ex: \texttt{pip install wbgapi})
  \item \texttt{affinity}: Dicionário com pontuações de 0--1 para cada pipeline
  \item \texttt{ecosystem\_integration}: Nome do pipeline principal
  \item \texttt{requires\_python}: Versão mínima do Python
  \item \texttt{license}: Tipo de licença (MIT, GPLv3, BSD, etc.)
\end{itemize}

\subsubsection{Matriz de Afinidade}

A matriz de afinidade $A_{ij} \in [0,1]$ quantifica a relevância da biblioteca $i$ para o pipeline $j$. Por exemplo:

\begin{itemize}[itemsep=1pt]
  \item $A(\text{mcp}, \text{MCP\_server}) = 1.00$ --- o SDK oficial do protocolo MCP é essencial para qualquer servidor MCP
  \item $A(\text{scholarly}, \text{SEEKER}) = 0.95$ --- a API do Google Scholar é altamente relevante para pesquisa bibliográfica
  \item $A(\text{wbgapi}, \text{data\_analysis}) = 0.95$ --- os indicadores do Banco Mundial são fundamentais para análise de dados socioeconômicos
\end{itemize}

\subsubsection{CLI com 7 Comandos}

O PyPI Scout oferece uma interface de linha de comando:

\begin{verbatim}
  search <termo>      Busca no PyPI e compara com catálogo
  catalog              Lista catálogo curado completo
  category <nome>      Filtra por categoria com afinidade
  install <pacote>     Instala via pip
  recommend <pipeline> Recomenda para SEEKER, MASWOS, etc.
  diff <pacote>        Compara versão catálogo x PyPI ao vivo
  help                 Ajuda de uso
\end{verbatim}

\section{Alteração 2: DataOrchestrator --- Acesso Universal a Dados}
% ══════════════════════════════════════════════════════════════════════════

\subsection{O Que É (Nível Introdutório)}

O \textbf{DataOrchestrator} é como um ``Google Tradutor'' para dados: você faz uma pergunta em português (``Qual o PIB do Brasil em 2023?''), e ele automaticamente descobre qual site tem essa informação, como acessá-la, e te entrega o resultado --- tudo sem você precisar saber que a fonte é o Banco Mundial, que a biblioteca é \texttt{wbgapi}, ou que o código do indicador é \texttt{NY.GDP.PCAP.CD}.

\subsection{Por Que Foi Feito (Nível Intermediário)}

O economista Herbert Simon recebeu o Prêmio Nobel em 1978 por demonstrar que seres humanos operam sob \textbf{racionalidade limitada}: tomamos decisões com informação e capacidade computacional finitas (SIMON, 1955). No contexto de pesquisa, isso significa que um pesquisador não deveria precisar aprender dezenas de APIs diferentes apenas para acessar dados --- essa carga cognitiva deveria ser assumida pelo sistema.

O DataOrchestrator implementa este princípio através de uma arquitetura de \textbf{3 camadas}:

\begin{enumerate}[itemsep=2pt]
  \item \textbf{Camada 1 --- Interface}: Você fala português. Ex: ``preço da ação da Apple''.
  \item \textbf{Camada 2 --- Roteamento}: O sistema analisa sua frase, identifica que ``ação'' e ``preço'' são palavras-chave do domínio financeiro, e seleciona a biblioteca \texttt{yfinance}. Se essa falhar, tenta \texttt{fredapi}. Se ainda falhar, tenta \texttt{wbgapi}.
  \item \textbf{Camada 3 --- Execução}: A biblioteca selecionada busca os dados na API externa (Yahoo Finance, neste caso) e retorna o resultado em formato padronizado.
\end{enumerate}

\subsection{Como Foi Implementado (Nível Avançado)}

\section{Alteração 3: Sistema de Auditoria Acadêmica Caixa Branca}
% ══════════════════════════════════════════════════════════════════════════

\subsection{O Que É (Nível Introdutório)}

Imagine que você está lendo um artigo científico e encontra a afirmação: ``O investimento em P\&D está correlacionado com inovação ($r = 0,73$)''. Como você pode verificar se isso é verdade? Você precisaria: (a) encontrar a fonte original citada; (b) verificar se a fonte realmente diz isso; (c) confirmar que os dados não foram manipulados.

O \textbf{Sistema de Auditoria Caixa Branca} faz exatamente isso, automaticamente. Toda interação do pesquisador com o ecossistema é registrada em um ``diário de bordo'' digital imutável (ninguém pode alterar depois). Cada afirmação no manuscrito é vinculada à sua evidência original. Cada evidência é vinculada à sua fonte verificável (DOI, arXiv). E um ``selo digital'' (hash SHA-256) garante que nada foi adulterado.

\subsection{Por Que Foi Feito (Nível Intermediário)}

O filósofo Karl Popper estabeleceu que uma teoria só é científica se for \textbf{falseável} --- ou seja, se for possível provar que ela está errada através de evidências (POPPER, 1959). No contexto de artigos gerados com auxílio de IA, isso significa que cada afirmação deve poder ser verificada independentemente por qualquer pessoa --- um revisor, um orientador, uma banca examinadora.

O cientista político Robert Axelrod demonstrou, através de torneios computacionais, que a estratégia ``Tit-for-Tat'' (coopere no primeiro movimento, depois replique o último movimento do oponente) é evolutivamente estável (AXELROD, 1984). Este princípio fundamenta o \textbf{protocolo TSAC} (87 palavras banidas): se o sistema detecta padrões típicos de IA (como uso excessivo de ``crucial'', ``essencialmente'', ``notavelmente''), ele ``pune'' o parágrafo reduzindo seu score de qualidade, incentivando o pesquisador a refinar o texto.

\subsection{Como Foi Implementado (Nível Avançado)}

\subsubsection{Arquitetura de 9 Componentes}

\begin{enumerate}[itemsep=2pt]
  \item \textbf{InteractionLogger} (337 linhas) --- O ``diário de bordo''. Singleton thread-safe que registra cada interação em formato JSONL (uma linha JSON por evento). Cada registro possui um hash SHA-256 calculado a partir do conteúdo --- se alguém tentar alterar o arquivo depois, o hash não confere. Registra 7 tipos de eventos:
  \begin{itemize}
    \item \texttt{session\_start}: Quando o pesquisador inicia uma sessão
    \item \texttt{interaction}: Cada pergunta e resposta
    \item \texttt{decision}: Cada decisão do pipeline (ex: ``usar arXiv como fonte primária'')
    \item \texttt{artifact}: Cada arquivo gerado (artigo, figura, tabela)
    \item \texttt{error}: Cada erro encontrado
    \item \texttt{paradigm\_set}: Quando o paradigma epistemológico é alterado
    \item \texttt{session\_end}: Quando a sessão é encerrada
  \end{itemize}

  \item \textbf{AcademicAuditTrail} (311 linhas) --- A ``trilha de migalhas''. Para cada parágrafo do manuscrito, registra quais evidências o sustentam e quais fontes (DOI/arXiv) as originaram. Gera relatórios em 3 formatos (Markdown para leitura humana, JSON para processamento automático, LaTeX para inclusão no artigo).

  \item \textbf{TokenEconomyMonitor} (220 linhas) --- O ``medidor de consumo''. Monitora quantos tokens foram gastos e compara com o orçamento do nível de publicação. Para N1 (Qualis A1), o orçamento é de 500.000 tokens --- suficiente para processar o equivalente a 2,5 livros de 300 páginas.

  \item \textbf{AuditInstrumentor} (293 linhas) --- O ``piloto automático''. Um wrapper que intercepta automaticamente todas as chamadas ao DataOrchestrator, registrando cada consulta sem que o pesquisador precise fazer nada.

  \item \textbf{ResearcherScore} --- A ``nota de qualidade''. Calcula um score de 0 a 100 baseado em 6 critérios:
  \begin{itemize}
    \item Densidade de evidências (25\% do score): Idealmente, cada parágrafo deve ter pelo menos 2 evidências.
    \item Fontes verificadas (20\%): Porcentagem de DOIs que foram confirmados via CrossRef.
    \item TSAC compliance (20\%): Porcentagem de parágrafos sem palavras banidas.
    \item Diversidade de fontes (15\%): Número de fontes únicas em relação ao número de parágrafos.
    \item Cobertura multi-domínio (10\%): Quantos dos 8 domínios de dados foram consultados.
    \item Peer review (10\%): Porcentagem de parágrafos revisados por pares.
  \end{itemize}
  A nota final segue o sistema de conceitos: A ($\geq$90), B ($\geq$75), C ($\geq$60), D ($\geq$40), F ($<$40).

  \item \textbf{BudgetAlert} --- O ``alerta de gastos''. Dispara avisos quando o consumo de tokens atinge 50\% (amarelo), 80\% (laranja) ou 90\% (vermelho) do orçamento.

  \item \textbf{AuditDashboard} --- O ``painel de controle''. Uma página HTML interativa (5,6KB) que mostra: score de qualidade, barra de progresso de tokens, alertas ativos, resumo da sessão e mapa de evidências.

  \item \textbf{AuditSearch} --- O ``buscador de histórico''. Permite encontrar sessões passadas por paradigma, nível, data ou score, e comparar múltiplas sessões lado a lado.

  \item \textbf{PipelineIntegration} --- O ``integrador''. Conecta automaticamente SEEKER $\rightarrow$ MASWOS $\rightarrow$ Auditoria, gerando relatório completo pronto para apêndice Qualis A1.
\end{enumerate}

\subsubsection{Protocolo TSAC: Formalismo}

O protocolo TSAC (\textit{Textual Science Audit Compliance}) monitora 87 palavras e expressões. Para um parágrafo $p$, a função de penalidade é:

\begin{equation}
  T(p) = \sum_{k=1}^{87} f(w_k, p)
\end{equation}

onde $f(w_k, p) = 1$ se a $k$-ésima palavra banida $w_k$ ocorre em $p$ (ignorando maiúsculas/minúsculas), e $f(w_k, p) = 0$ caso contrário. Um parágrafo é ``limpo'' se $T(p) = 0$.

\section{Alteração 4: Reasoning Orchestrator v9.0 + Teoria dos Jogos}
% ══════════════════════════════════════════════════════════════════════════

\subsection{O Que É (Nível Introdutório)}

A \textbf{Teoria dos Jogos} é um ramo da matemática que estuda como pessoas (ou agentes de IA) tomam decisões quando o resultado depende não apenas da sua própria escolha, mas também das escolhas dos outros. O exemplo mais famoso é o \textbf{Dilema do Prisioneiro}: dois suspeitos são presos separadamente. Se ambos cooperarem (ficarem em silêncio), pegam 1 ano cada. Se ambos traírem (dedurarem o outro), pegam 3 anos cada. Se um cooperar e o outro trair, o traidor sai livre e o cooperador pega 5 anos. O que você faria?

O \textbf{Reasoning Orchestrator v9.0} incorpora 10 dessas estratégias da Teoria dos Jogos ao raciocínio do ecossistema. Isso significa que, quando o sistema precisa decidir entre diferentes abordagens --- por exemplo, qual fonte de dados usar, ou como estruturar um argumento --- ele aplica o mesmo rigor matemático que ganhou Prêmios Nobel de Economia para Nash (1994), Harsanyi (1994) e Shapley (2012).

\subsection{Por Que Foi Feito (Nível Intermediário)}

Agentes de IA que interagem entre si --- como os 125 agentes do OpenCode Ecosystem --- enfrentam exatamente o mesmo tipo de dilemas estratégicos que a Teoria dos Jogos modela. Quando o SEEKER precisa decidir se confia nos resultados do Semantic Scholar ou prefere buscar no arXiv, ele está essencialmente jogando um jogo onde a ``jogada'' é a escolha da fonte.

Sem a Teoria dos Jogos, o sistema usaria heurísticas simples (``sempre use arXiv primeiro''). Com a Teoria dos Jogos, ele pode modelar formalmente a situação: o Semantic Scholar é um ``jogador'' que pode ter informações incompletas (HARSANYI, 1967), múltiplas fontes podem formar ``coalizões'' cuja contribuição marginal pode ser calculada (SHAPLEY, 1953), e a estratégia ótima depende do que as outras fontes estão fazendo (NASH, 1950).

\subsection{Como Foi Implementado (Nível Avançado)}

\subsubsection{Fundamentação Matemática}

Um jogo na forma normal é definido pela tripla:

\begin{equation}
  \Gamma = (N, \{S_i\}_{i \in N}, \{u_i\}_{i \in N})
\end{equation}

onde:
\begin{itemize}[itemsep=1pt]
  \item $N = \{1, \ldots, n\}$ é o conjunto de jogadores (ex: 125 agentes)
  \item $S_i$ é o conjunto de estratégias disponíveis para o jogador $i$
  \item $u_i: \prod_{j \in N} S_j \to \mathbb{R}$ é a função de utilidade (``recompensa'') do jogador $i$
\end{itemize}

\subsubsection{Equilíbrio de Nash}

Um perfil de estratégias $s^* = (s_1^*, \ldots, s_n^*)$ é um Equilíbrio de Nash (NASH, 1950, DOI: 10.1073/pnas.36.1.48) se:

\begin{equation}
  \forall i \in N, \forall s_i \in S_i: u_i(s_i^*, s_{-i}^*) \geq u_i(s_i, s_{-i}^*)
\end{equation}

Em palavras: nenhum jogador pode melhorar seu resultado mudando unilateralmente de estratégia. No ecossistema, isso significa que a configuração de agentes é ``estável'' --- ninguém tem incentivo para ``trapacear''.

\subsubsection{Valor de Shapley}

O valor de Shapley (SHAPLEY, 1953, DOI: 10.1515/9781400881970-018) mede a contribuição marginal de cada jogador para cada coalizão possível:

\begin{equation}
  \phi_i(v) = \sum_{S \subseteq N \setminus \{i\}} \frac{|S|!(n-|S|-1)!}{n!}[v(S \cup \{i\}) - v(S)]
\end{equation}

No ecossistema, isso é usado para atribuir ``crédito'' a cada fonte de dados: quanto o arXiv contribuiu para o resultado final, em comparação com o Semantic Scholar?

\subsubsection{10 Estratégias Integradas}

\begin{enumerate}[itemsep=1pt]
  \item \textbf{Equilíbrio de Nash} (NASH, 1950) --- Estratégia ótima não-cooperativa. Aplicação: otimização de comportamento em sistemas multiagente.
  \item \textbf{Dilema do Prisioneiro} --- Cooperação vs traição. AXELROD (1984) demonstrou estabilidade evolutiva do Tit-for-Tat.
  \item \textbf{Soma Zero} --- Competição pura. VON NEUMANN \& MORGENSTERN (1944).
  \item \textbf{Tit-for-Tat} (AXELROD, 1984) --- Cooperação condicional iterada.
  \item \textbf{Stackelberg} --- Vantagem do primeiro movimento.
  \item \textbf{Barganha de Nash} --- Solução cooperativa.
  \item \textbf{Sinalização} (SPENCE, 1973) --- Informação assimétrica.
  \item \textbf{Evolutivo} (SMITH e PRICE, 1973) --- Seleção natural de estratégias.
  \item \textbf{Bayesiano (Harsanyi)} (HARSANYI, 1967) --- Crenças sobre tipos incertos.
  \item \textbf{Cooperativo (Shapley)} (SHAPLEY, 1953) --- Valor marginal.
\end{enumerate}

\subsubsection{Para o Leitor Iniciante: Exemplo do Cotidiano}

Pense na Teoria dos Jogos como as ``regras do xadrez'' para agentes de IA. No xadrez, você não decide seu movimento pensando apenas no que você quer fazer --- você pensa no que seu oponente fará em resposta. O Equilíbrio de Nash é como uma posição de xadrez onde nenhum jogador pode melhorar sua situação mexendo apenas uma peça. O Tit-for-Tat é como jogar xadrez copiando o último movimento do oponente: se ele foi agressivo, você também é; se ele foi defensivo, você também é.

\section{Discussão}
% ══════════════════════════════════════════════════════════════════════════

\subsection{Síntese das Contribuições}

As quatro alterações documentadas formam um ecossistema coeso com uma progressão lógica:

\begin{enumerate}[itemsep=2pt]
  \item \textbf{PyPI Scout} resolve o problema de \textbf{descoberta}: como encontrar a biblioteca certa entre 500.000 opções.
  \item \textbf{DataOrchestrator} resolve o problema de \textbf{acesso}: como consultar 8 domínios de dados sem aprender 8 APIs diferentes.
  \item \textbf{Sistema de Auditoria} resolve o problema de \textbf{confiança}: como provar que cada afirmação é verdadeira e rastreável.
  \item \textbf{Reasoning Orchestrator} resolve o problema de \textbf{rigor}: como garantir que 68 tipos de raciocínio --- incluindo 10 da Teoria dos Jogos --- sejam aplicados a cada decisão.
\end{enumerate}

\subsection{Limitações}

\begin{enumerate}[itemsep=1pt]
  \item O parser \texttt{QueryIntent} usa casamento de palavras-chave. Para consultas ambíguas (ex: ``Apple'' pode ser a empresa ou a fruta), a acurácia é limitada.
  \item A biblioteca \texttt{scihub-cn} não pôde ser instalada devido a incompatibilidade de build com Python 3.12.
  \item O ResearcherScore mede qualidade de \textit{processo}, não de \textit{conteúdo}. Um artigo pode ter score 100 no processo e ainda assim ser conceitualmente fraco.
\end{enumerate}

\subsection{Trabalhos Futuros}

\begin{enumerate}[itemsep=1pt]
  \item Integrar LLM local para desambiguação semântica no QueryIntent.
  \item Cache distribuído para reduzir latência de APIs externas.
  \item Streaming de dados em tempo real via WebSocket.
  \item Expansão Qualis A1 para Ciências Humanas e Exatas.
  \item Métricas de qualidade de conteúdo para o ResearcherScore.
\end{enumerate}
